# RAI Ops Compliance Report Guide

This document explains how to interpret the various sections of the Compliance Reports generated by the RAI Ops platform.

## 1. Report Types

The platform generates two primary types of reports:
1.  **Compliance Summary Report**: A high-level audit trail of system activities, user actions, and security events.
2.  **Red Team Test Report**: A detailed technical breakdown of a specific adversarial attack simulation.

---

## 2. Compliance Summary Report

This report provides evidence of governance and operational oversight. It is typically used for internal audits (SOC2, ISO 42001).

### Key Columns

| Column | Description |
| :--- | :--- |
| **Timestamp** | The UTC date and time when the action occurred. Essential for establishing a timeline of events. |
| **Action** | The specific operation performed (e.g., `CREATE_TEST`, `VULN_STATUS_CHANGE`, `LOGIN`). |
| **Resource** | The system entity affected by the action (e.g., `test/15`, `user/admin`). |
| **Status** | The outcome of the action (`SUCCESS`, `FAILURE`, `DENIED`). |

### Common Log Patterns
- **VULN_STATUS_CHANGE**: Indicates that a human reviewer has triaged a vulnerability found by the Red Team (e.g., moving it from "New" to "Resolved"). This proves human-in-the-loop oversight.
- **SCHEDULED_TEST_RUN**: Automated background tests. Frequent failures here may indicate system instability or persistent model issues.

---

## 3. Red Team Test Report

This report details the results of a specific security test session against an AI model.

### Header Information
- **Test Name**: The identifier for the test session.
- **Target System**: The specific model endpoint or application being tested.
- **Test Type**: The attack vector used (e.g., *Prompt Injection*, *PII Leakage*).

### Results Summary

#### Risk Level
Calculated based on the severity and success rate of attacks.
- **Critical**: The model succumbs to trivial attacks, exposing sensitive data or control. Immediate remediation required.
- **High**: Complex attacks succeed. Mitigation is required before public deployment.
- **Medium**: Edge cases found, but core safety guardrails held. Monitor closely.
- **Low**: The model resisted most adversarial attempts.

#### Overall Score
A normalized score (0-100%) representing the model's resilience.
- **90-100%**: Excellent Robustness
- **70-89%**: Acceptable for low-risk applications
- **<70%**: Unsafe for deployment

#### Vulnerabilities Found
This section lists specific successful exploits.

**Example Entry:**
> **Type**: Prompt Injection
> **Severity**: High
> **Description**: "Model revealed system instructions via adversarial prompt."

*Interpretation*: The attacker successfully bypassed the system prompt (e.g., "You are a helpful assistant") and forced the model to reveal its internal logic or instructions.

---

## 4. Operational Monitoring (Drift)

While not always in the PDF, compliance officers should review the Monitoring dashboard for **Drift**.

- **Baseline**: The performance metric (accuracy, fairness) at the time of deployment.
- **Current**: The real-time metric based on recent traffic.
- **Drift Score**: The percentage deviation.

> **Rule of Thumb**: A drift score >10% usually warrants investigation. A score >20% (or fairness deviation >5%) typically triggers an automated alert in the system.
